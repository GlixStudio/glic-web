GLIC - GLitch Image Codec

Document contains all codec internals with all of its glitch sweet spots. If you feel that it’s against glitch ideas (like “glitch is a matter of chance”), read till “Encoding process” section and skip the rest of the file. You can look through to see example images.

You were warned :)

Facebook group: https://www.facebook.com/groups/GlitchImageCodec/

GLIC is an image codec designed especially for artistic databending purposes. It includes steps and techniques used in most modern codecs like h264/h265(hevc) or VP9. Therefore all of known glitch artifacts are easy to reveal plus plenty of others. Codec is designed to give more control during databending and a possibility to easily turn on and off features and encoding steps.

There are three main issues which are problematic in widely used codecs:

  1. Structure and data tangled together
  2. Entropy (binary coding) - responsible for final compression
  3. Integrity checks

That’s why you can encounter problems with decompression after databend (errors, noise, artifacts only, etc.).

Test image source used in this document: https://stocksnap.io/photo/8MX9Y2ESWB

Source code: https://github.com/GlitchCodec/GLIC

Put content of the repository in the ‘GLIC’ folder.

Codec is written in Processing (works well with Processing 2 and 3) + ControlP5 library (Sketch -> Import Library... -> Add Library… -> ControlP5)

Don’t forget to add more available RAM to your Processing (File -> Preferences -> Increase…). Set it to half of your RAM.

Design goals

  1. Metadata, structure and image data separation (6 different blocks of data)
  2. Reveal artifacts without losing image context / structure
  3. Separation of encoding steps (with option to turn on/off):
  1. Color space conversion
  2. Adaptive segmentation
  3. Predictions and residuals
  4. Quantization
  5. Wavelet transformations with compression
  6. Entropy coding / binary packing
  1. Binary format format prepared for hexediting - fixed position in header
  2. Plenty of possibilities for each step:
  1. 16 color spaces
  2. Quadtree segmentation based on image structure with variable block sizes - from 2 to 512 pixels
  3. 13 simple prediction modes + two configurable meta predictions: angle and reference
  4. 2 ways to encode prediction residuals
  5. 67 wavelets + 2 types of transformations -> 2 * 67 variants
  6. currently no entropy coding (but 2 planned: packed RAW and RLE)
  1. Extensible format - possibility to add another kind of encoding for each step
  2. Color space channels separation
  3. (Almost) no errors during decoding invalid data - you can always get some result
  4. No integrity checks
  5. Lossy and lossless
  6. GUI
  7. Batch processing

Additional notes:

  1. This is not a format to compress your images, sorry.
  2. No alpha channel (can be added if necessary)
  3. Sometimes you can get instant glitch or artifacts on the right side and bottom - if you don’t want them, the image dimension should be power of 2 (2, 4, 8, 16, 32, 64, 128, 256, 512, 1024, 2048, 4096, etc…)
  4. It’s slow and can eat a lot of memory.
  5. No deblocking / in-loop filter - sharp block edges

Comparison

So what is used in known and existing codecs?

  1. As mentioned - segmentation, metadata, image data are usually packed together and it’s hard to find a sweet spot in the data during hexediting.
  2. Only three color spaces are mainly used: (s)RGB (png), YUV (video codecs, webp), YCbCr (jpeg)
  3. Fixed segmentation sizes for coded blocks: 1x1 pixels (png), 8x8 pixels (jpeg, webp chroma), 4x4/16x16 (webp luma)
  4. Limited number of prediction modes: 5 (png), 5+1 (webp), 0 (jpeg), 2+1 (hevc)
  5. Usually one transformation like DCT (jpeg, hevc) or some wavelet (dirac, snow)

Encoding process

Here I describe what are steps in detail and how to configure them during encoding. I mark also points where you can get instant glitch or low quality artifacts.

Color space

Image pixels are stored as data about color. Color is just: 3 numbers which values are between 0 and 255 (sometimes there is another number which describes opacity - alpha, or blackness in CMYK). Which triple is which color is defined as color space. You probably know plenty of them like (s)RGB, CMY, HSB, YUV, YCbCr, etc. The same triple is different in each color space.

In GLIC you can select one from 16 color spaces.

We have 3 types of colors paces:

  • luma based, one channel describes luma (brightness), other two hue: OHTA, YXY, XYZ, LUV, LAB, RGGBG, YPbPr, YCbCr, YDbDr, GS (only luma), YUV
  • hue based, one channel describes hue, the rest is intensity: HSB, HCL, HWB

Channels

Next step is channel separation. All color space channels are separated and are treated independently. This gives you the opportunity to apply different rules for every channel.

Segmentation

Each channel is then cut into squares of pixels. And what is most important is that this structure keeps the structure of the image. The idea is to use small squares for dense areas and larger squares for flat/similar color areas.

Algorithm chooses randomly 10% of block points and calculates standard deviation if it’s higher than threshold parameter, block is subdivided into 4 smaller squares.

Visualization of segmentation in YXY color space:

or in HWB color space

Configuration of segmentation is based on 3 variables:

  1. MIN - minimum block size (power of 2)
  2. MAX - maximum block size (power of 2)
  3. THR - values from 5 to 50, lower number - better precision, higher - worse.

When you use transformations you can get artifacts at the bottom or right side of the image. The reasons are:

  1. Image size is not divisible by 2^MIN (example, MIN = 4, your image dimensions should be divisible by 2^4=16). Adjust image size or MIN value.
  2. THR is too big. Lower it (15-20 is optimal)
  3. Color outside (see below) is similar to color at boundaries. Set color outside to contrasting values 0,0,0 or 255,255,255

Predictions

And we are at the heart of modern codecs. Prediction is a technique used to guess pixel values from surrounding (already known) pixels. In real codecs the main reason is to get as much zeroes as possible. Zeroes compress best.

Process goes like that:

  1. Select prediction algorithm
  2. Guess pixels in the block using pixels from top and left side (they are already guessed)
  3. Subtract guessed pixels from original pixels (this is named residual) and keep for further processing.

When you choose wisely you can get plenty of zeroes.

In GLIC I’ve implemented 15 algorithms including two special (angle based and reference based).

Color around the image

One issue related to prediction is what happens on image boundaries. You have to define what color is outside to have prediction work better.

It’s visible on almost all above images as gray squares on the left (top) sides of the image.

Clamp method

When residuals from the previous step are calculated values are from -255 to 255 and they can be left like that or encoded. Currently there are 2 options:

(De)Quantization

At this stage data are quantized. It’s made by simple integer division by a number you set in “quantization value” parameter’. 0 means - no quantization

Setting high value gives instant glitch (low quality). Maximum value is 255.

During decoding data are dequantized by multiplication.

Transformation

Next step is to transform blocks and possibly compress using wavelet transformations. There are plenty of them.

Transformation and wavelet is chosen for the channel globally.

Additionally you can compress transformation which gives quality loss. Compression value is from 0 (no compression) to 255 (highest compression).

Another compression factor is scale.

Transformations give values from -1 to 1, to save it as integer I have to multiply it by some value. Lower value gives worse quality.

Internals

ENCODING_RAW

Each value is stored as 4 byte integer (bigendian)

ENCODING_PACKED

if transformation is not used:

  • 8 bits for CLAMP_MOD256 (values 0-255)
  • 9 bits for CLAMP_NONE (values -255 - 255)

if transformation is use, number of bits determined by scaling value (+1 for CLAMP_NONE)

ENCODING_RLE

First bit determines if value is repeated or not:

  • repeated: ‘1’ + 7 bits (up to 129 repetitions) followed by value the same as in ENCODING_PACKED
  • singular: ‘0’ + value

GLIC File Format

All encoded data and configuration is stored in the glic file. Structure of the file goes as below. See details and examples in ‘Decoding and databending’ section.

Headers

Below you have described the meaning of the GLIC file viewed in hex editor.

There are 6 sections:

  1. First header - image size, color space and color outside image (RGB)
  2. Data sizes - sizes of 3 data blocks for separate for each channel:
  1. Segmentation data
  2. Prediction additional info (angle and ref)
  3. Image data
  1. Second header, for each channel configuration:
  1. Global prediction method (if 0x00, prediction additional info is used)
  2. Quantization value
  3. Clamp method
  4. Wavelet
  5. Transformation type
  6. Transformation scaling value
  7. Final encoding method
  1. Data block with information about segmentations (3 channels)
  2. Data block with information about local prediction method (prediction method, angle for PRED_ANGLE and reference data for PRED_REF) (3 channels)
  3. Data block for image data (3 channels)

If a section stores channel data, each channel section is marked with ‘CH01’, ‘CH02’ or ‘CH03’ (used in section 3-6).

First header

Yellow left - image width

Yellow right - image height

Red - color space (can be changed from GUI)

Green - color outside (r,g,b) (can be changed from GUI)

Data sizes

Yellow line - size of segmentation data

Blue line - size of prediction additional information data

Pink line - size of image data

Column 1 (red dot) - Channel 1

Column 2 (green dot) - Channel 2

Column 3 (blue dot) - Channel 3

Second header

For each channel marked as ‘CH01’, ‘CH02’ or ‘CH03’ you can set following configuration:

  1. Global prediction method
  2. Dequantization value
  3. Clamp method
  4. Wavelet
  5. Transformation type
  6. Scaling value
  7. Final encoding method

All can be changed from GUI

Data

The rest of the file stores data:

Segmentation info (quadtree), marked with word: “SEGMENTATION”

Followed by 512 FF

Prediction data, each marked block (5 numbers) describes one segment. Marked with word “PREDICTDATA”

  1. Local prediction method (here PRED_NONE). If set to any value, replaces global prediction method set in second header
  2. X position for PRED_REF
  3. Y position for PRED_REF
  4. Quarter for PRED_ANGLE
  5. Angle for PRED_ANGLE

Block is followed by 512 FF

Finally image data encoded by encode method.
